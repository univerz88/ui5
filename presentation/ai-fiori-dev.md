## 1. 배경과 문제 인식

### 상황

SAP ERP 구축 프로젝트에는 수백 개의 개발 프로그램이 존재한다.  
각 프로그램의 설계 완료일, 개발 계획일, 담당자, 공수(M/D) 등의 데이터는  
**엑셀 파일로 관리**되고 있었다.

### 문제

엑셀은 데이터를 **저장**하는 데는 적합하지만, 아래 질문에 즉시 답하기가 어렵다.

| 질문 | 엑셀의 한계 |
|------|------------|
| 지금 전체 개발 완료율이 얼마나 되나? | 수식 계산 후 수동 확인 |
| 마감 위험 프로그램이 몇 개인가? | 조건부 서식 + 필터링 |
| 구분(CA/HQ/VN)별 진척이 어떻게 다른가? | 피벗 테이블 별도 생성 |
| 설계 완료 후 아직 개발 안 된 것은? | 복잡한 필터 조합 |

> **핵심 문제**: 데이터는 있는데, 현황을 **한눈에 파악**할 수 없다.

## 2. 해결 방향 설정

### 목표

엑셀 파일을 **그대로 유지**하면서, 업로드만 하면 즉시 현황을 볼 수 있는  
**대시보드**를 만들자.

### 요구사항 정의 (Claude와 함께)

```
요구사항을 Claude에게 설명한 내용:

"SD/MM/PP/FI/CO/CM 모듈별로 개발 프로그램 현황이 있어요.
 각 프로그램에는 설계완료일, 개발계획일, 담당자, M/D가 있고
 구분은 CA(공통)/HQ(본사)/VN(베트남)으로 나뉩니다.
 마감은 2026-05-31이고, 대시보드로 만들고 싶어요."
```

Claude는 이 설명을 바탕으로 아래를 함께 정의했다.

- **필요한 KPI**: 전체 프로그램 수, 설계완료, 개발완료, 잔여 M/D, D-Day
- **시각화**: 구분별 완료/잔여 차트, M/D 도넛, 상태별 분포
- **상태 판단 기준**: 마감일 기준 가용일수 vs 필요공수 비교
- **드릴다운**: 설계완료 후 미개발, 설계 미완료 목록

## 3. AI와 함께 기획하기

### 컨셉 → 데이터 구조

Claude와의 대화를 통해 Excel 컬럼 구조를 분석하고 파싱 로직을 설계했다.

```javascript
// 주요 컬럼 정의 (BASE_C, 0-based 인덱스)
const BASE_C = {
  mod:      1,   // B열 — 모듈명 (SD/MM/PP...)
  zone:     4,   // E열 — 구분 (CA/HQ/VN)
  name:     6,   // G열 — 프로그램명
  designA:  21,  // V열 — 설계 실제일 (완료 여부 판단)
  md:       23,  // X열 — 개발 공수 M/D
  devPlan:  25,  // Z열 — 개발 계획일
  devAct:   26,  // AA열 — 개발 실제일 (완료 여부 판단)
};
```

### 상태 판단 로직

```
가용 영업일수 (오늘 ~ 개발계획일) vs 필요 공수 (M/D)

가용일 < M/D      → 마감초과 (빨강)
계획일 > 5/14     → 위험    (주황)
계획일 있음       → 적정    (초록)
계획일 없음       → 일정미정 (회색)
```

> 이 로직을 Claude에게 설명하고, 코드로 구현하게 했다.  
> "틀렸다"고 수정 요청하면 즉시 반영되는 방식으로 진행했다.

## 4. 프로토타입 제작 (demo.html)

### 방식

SAP UI5 개발 전, **HTML 단일 파일**로 먼저 만들었다.

- 서버 불필요 — 파일을 더블클릭하면 바로 실행
- 실제 엑셀 데이터를 업로드해서 즉시 검증 가능
- UI/UX 피드백을 빠르게 반영

### 주요 기능

| 기능 | 구현 내용 |
|------|----------|
| 엑셀 업로드 | SheetJS로 브라우저 내 파싱 |
| 모듈 탭 | ALL / SD / MM / PP / FI / CO / CM |
| KPI 카드 | 6개 — 전체/설계완료/개발완료/잔여/M/D/D-Day |
| 진행률 바 | 전체 + 구분별 (CA/HQ/VN) |
| 차트 3개 | 구분별 완료/잔여 / M/D 도넛 / 상태별 분포 |
| 잔여 테이블 | 공수 부하율 바 포함 |
| 드릴다운 모달 | 설계완료 후 미개발 / 설계 미완료 목록 |

### 개발 과정

```
1. Claude에게 기능 설명
2. Claude가 HTML/CSS/JS 생성
3. 실제 엑셀로 테스트
4. "이 부분이 이상하다" → Claude가 수정
5. 반복
```

프로토타입 완성까지 **실제 코딩 없이** 진행되었다.

## 5. SAP Fiori App으로 전환

### 왜 전환했나?

| 항목 | demo.html | Fiori App |
|------|----------|-----------|
| SAP 표준 UI | ✗ | ✓ (SAP Horizon 테마) |
| Launchpad 통합 | ✗ | ✓ |
| 유지보수 | 어려움 | SAP 표준 방식 |
| 타입 안전성 | ✗ | ✓ (TypeScript) |

### 활용한 도구: MCP (Model Context Protocol)

Claude Code에 UI5 MCP 서버를 연결하면, Claude가 SAP UI5 전문 지식을 자동으로 참조한다.

```
MCP 도구 활용 흐름:

get_guidelines        ← UI5 개발 원칙 확인
create_ui5_app        ← TypeScript 프로젝트 생성
get_api_reference     ← VizFrame, FileUploader API 확인
run_ui5_linter        ← 코드 품질 검사
run_manifest_validation ← 설정 파일 검증
```

### 핵심 기술 스택

| 항목 | 내용 |
|------|------|
| 프레임워크 | SAPUI5 1.146.0 |
| 언어 | TypeScript |
| 차트 | sap.viz (VizFrame) |
| 엑셀 파싱 | xlsx npm + ui5-tooling-modules |
| 테마 | SAP Horizon |

### UI5 경험이 없어도 가능했던 이유

1. **MCP가 가이드라인을 자동 제공** — Claude가 최신 UI5 문서를 참조
2. **오류 발생 시 Claude가 원인 설명** — 단순 코드 수정이 아니라 이해와 함께 진행
3. **프로토타입이 명세서 역할** — demo.html이 있어 요구사항이 명확했음

## 6. 결과 및 시사점

### 결과

- 엑셀 업로드 → 즉시 대시보드 시각화 실현
- SAP Fiori 표준을 준수하는 TypeScript 앱 완성
- UI5 사전 경험 없이 MCP + Claude로 개발 완료

### AI 협업 개발의 특징

**잘 되는 것**
- 반복적인 코드 생성 및 수정
- API 문서 기반 구현 (MCP 활용)
- 프로토타입 → 구조화 변환

**주의할 것**
- 방향 설정과 요구사항 정의는 사람이 해야 함
- "무엇을 만들지"를 명확히 할수록 결과가 좋아짐
- 생성된 코드는 반드시 검증 필요

### 새로운 개발 패러다임 제안

```
기존: 요구사항 → 설계 → 개발 → 테스트 (순차적, 수주~수개월)

AI 협업:
  요구사항 → 프로토타입 (당일)
           → 검증/피드백 (즉시)
           → 실제 앱 전환 (수일)
```

> **AI는 개발 속도를 높이는 도구다.**  
> 무엇을 만들지 결정하는 것은 여전히 사람의 몫이다.

## 부록: 이 방식으로 시작하려면

### 필요한 것

- Claude Code (VS Code 확장 또는 CLI)
- UI5 MCP 서버 연결
- Node.js + UI5 Tooling

### 첫 번째 프로젝트 시작 방법

```bash
# 1. Claude Code에서 UI5 MCP 활용
get_guidelines      # 개발 원칙 확인
create_ui5_app      # 프로젝트 생성

# 2. 로컬 실행
cd my-fiori-app
npm start
# → http://localhost:8080/index.html
```

### 핵심 팁

- 프로토타입(HTML/스케치)을 먼저 만들어 요구사항을 확정하라
- Claude에게 "왜"를 설명하면 더 좋은 코드가 나온다
- MCP 도구를 적극 활용하면 UI5 초보도 표준을 지킬 수 있다
